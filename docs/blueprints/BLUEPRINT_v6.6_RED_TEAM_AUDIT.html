A. ARCHITECTURE RED TEAM

Question 1 CRITICAL
Context overflow: 50 orders on 128K model

The blueprint contradicts itself. It says "50 maximum per user" AND "smart-context sends 16K." But it also says "Test J: send ALL 50 orders with 1M context." Sarvam 105B has 128K context, not 1M. Only Gemini 2.5 Pro has 1M.

What actually happens if someone uploads 50 orders:

  • 50 Kannada DDLR orders × ~2,500 tokens each = ~125,000 tokens
  • Add system prompt (4K) + facts (2K) + thinking overhead = ~135K tokens → OVERFLOW on Sarvam 128K
  • Smart-context (16K) saves you, but only if it's ALWAYS used. The blueprint leaves the door open for full-context mode.
Fix: Hard-cap at 50 orders in storage, but NEVER send more than 5-8 in context. Smart-context is not optional — it's mandatory. Remove "full 50-order context" as an option for Sarvam. Test J should only test against Gemini/Claude which actually have 1M context.
Question 2 WARNING
Auto-prompt from mixed-style orders

If a caseworker uploads 20 orders from 3 different officers (which Banu likely will — he's collecting from colleagues), the auto-generated prompt will be a confused average of 3 styles.

The blueprint assumes "upload YOUR best orders" but real-world behavior will be "upload whatever I can find." There's no style-consistency check.

Fix: Add a clustering step before auto-prompt. Group orders by writing style (sentence length, vocabulary fingerprint, section ordering). If >1 cluster detected, ask user: "These seem to be from different writers. Use all, or pick one style?" This is a Phase 1B feature, not a blocker.
Question 3 CRITICAL
Corrections replace originals — no rollback

"Corrections replace originals in reference set" — this is destructive by design. If Banu makes 3 corrections and 1 is wrong, that wrong correction is now your training data. There is no version history, no undo, no way to know which correction degraded quality.

This is like a textbook with no errata process. Once the wrong answer is printed, every student learns it.

Fix: Never delete originals. Keep a version chain: Original → Correction v1 → Correction v2. Use the LATEST version for generation, but keep ALL versions. Add a "revert to original" button. This is 2 extra database columns — trivial to implement, catastrophic to skip.
Question 4 WARNING
Self-validation: AI judges its own work

The self-validation design (hold back 3 orders, generate from facts, compare vs originals) is clever but has a known bias: the same model that generates will also judge. LLMs consistently rate their own output higher than human evaluators do.

However, the blueprint's approach is better than pure self-rating because it compares against a REAL answer key (the original order). The comparison is structural (sections present, facts preserved), not subjective ("is this good?").

Fix: Acceptable for Phase 0-1. But track Banu's manual corrections as ground truth. If self-validation says 95% but Banu corrects 30% of orders, your self-validation is lying. Add a "validation accuracy" metric: % of times self-validation score matches Banu's actual satisfaction.

Question 7 WARNING
Sarvam 105B quality — 90/100 reliable across case types?

The 90/100 score was benchmarked on one case type (DDLR appeal orders, likely Hesaraghatta). No evidence it holds for mutations, corrections, suo motu, or review orders. Different order types have different structures, legal language, and section requirements.

90/100 on appeals could be 70/100 on mutations. We don't know.

Fix: Before locking Sarvam as default, test on at least 3 case types × 3 orders each = 9 orders minimum. Score each. If any type drops below 85, route that type to Sonnet automatically. This is a 2-hour test — do it before April 5.
Question 8 WARNING
python-docx with Kannada complex_script

Known issues confirmed by research:

  • python-docx can SET Kannada text, but font styling (bold, italic) may not apply correctly to non-ASCII Unicode characters
  • Conjunct characters (ottakshara) in Kannada may not render correctly in all viewers
  • The document looks fine in MS Word but may break in LibreOffice or Google Docs

Since your users are government officers using Windows + MS Word, this is probably OK. But test with the actual fonts (Tunga, Noto Sans Kannada) on a real government PC.

Fix: Get one DOCX file onto Banu's actual office computer and verify it opens correctly. If bold headers don't render, switch to fpdf2 for PDF output (better Unicode support). This is a 10-minute test — do it in Phase 0.
Question 9 WARNING
Sarvam Vision OCR — 100 pages/month free

For Phase 0 with 1 user (Banu): probably enough. Each case file is 2-5 pages. Banu testing 2-3 cases = ~15 pages. Well within 100.

But Phase 1 with 10 users: 10 users × 5 cases/month × 3 pages = 150 pages. Already over free tier. Professional tier is ₹2,999/month for 5,000 pages.

Fix: Phase 0 is fine. Budget ₹2,999/month for Sarvam Vision from Phase 1. Fallback: Google Cloud Vision (₹0.12/page, no free tier but pay-as-you-go). Add this to the cost model.

C. SCALE STRESS

Question 11 MANAGEABLE
50 users × 50 files = 2,500 files storage cost

Average DDLR order PDF: ~200KB. 2,500 files × 200KB = ~500MB. Supabase free tier gives 1GB storage. This is fine for Phase 0-1.

At 500 users: 25,000 files = ~5GB. Supabase Pro gives 100GB for $25/month. Still fine.

The real cost isn't storage — it's database egress. Supabase free tier limits data transfer to 2GB/month. Each AI call that reads 5 reference orders from storage = ~50KB. 500 users × 20 orders/month × 50KB = 500MB transfer. OK for now.

Fix: Not a problem until 200+ users. Monitor storage and transfer in Supabase dashboard.
Question 12 WARNING
Sarvam rate limits → profitability collapse

The entire profitability model rests on "Sarvam 105B is FREE." But Sarvam's free tier gives ₹1,000 in credits — not unlimited free usage. The Startup Program gives 6-12 months of free credits, but the exact amount is not publicly documented.

If Sarvam free credits run out or rate limits hit:

  • Sarvam Pro: ₹10,000/month (for 200 req/min)
  • All 4 packs go from "profitable" to "₹10K fixed cost eating margins"
  • At 50 users × 20 orders/month = 1,000 orders. Revenue: ~₹20,000. Sarvam cost: ₹10,000. Margin: 50% — survivable but tight.
  • At 20 users: Revenue: ~₹8,000. Sarvam cost: ₹10,000. Net loss.
Fix: Get Sarvam Startup Program credits CONFIRMED with exact numbers before building the entire cost model around "free." Have a fallback: Gemini 2.5 Flash (cheap, 1M context) or self-hosted Sarvam on GPU (complex but no per-call cost). Add a "break-even user count" to the pricing model.
Question 13 WARNING
Model deprecation risk

Sarvam 105B is an open-source model. Even if Sarvam (the company) discontinues the API, the model weights are available. You could self-host. But self-hosting a 105B parameter model requires serious GPU infrastructure (~₹50,000-80,000/month for A100 instances).

More realistic risk: Sarvam changes pricing tiers or rate limits. This has happened with every AI startup (OpenAI, Anthropic, Cohere all raised prices after initial free/cheap tiers).

Fix: The blueprint's "model-agnostic" principle is correct. Keep the abstraction layer. But also test Gemini 2.5 Flash and Llama 3.3 as backup models NOW, not "after Phase 0." A 2-hour benchmark today prevents a crisis later.

Question 18 CRITICAL
#1 tech decision that could be wrong

Rewriting from Streamlit to Next.js during a 10-day deadline.

You have a working 713-line Streamlit app. It does OCR, generation, DOCX output, 13 AI models. It works on localhost. Instead of deploying it, the blueprint says "throw it away and rebuild in Next.js in 3 days."

This is the classic "shiny new framework" trap. Streamlit is ugly? Yes. But ugly-and-shipped beats beautiful-and-unfinished. Every startup that died did so because they rebuilt instead of shipped.

If this decision is wrong, it costs you the April 5 deadline AND the working prototype.

Question 19
If rebuilding from scratch, one thing to change

Start with the correction feedback loop, not the generation pipeline.

The blueprint focuses on "generate orders." But the real product is "generate orders that get better over time." The correction → reference update → quality improvement loop is what creates the moat. Without it, you're just a ChatGPT wrapper with Kannada formatting.

If I rebuilt from scratch: Day 1-3 = simple generation (even with ChatGPT). Day 4-7 = correction tracking + reference versioning + quality scoring. Day 8-10 = self-improvement pipeline. The generation quality catches up. The learning loop is what competitors can't copy.

F. TOP 3 RECOMMENDATIONS

1. DON'T REWRITE — DEPLOY STREAMLIT NOW

Put the existing 713-line Streamlit app on a ₹500/month Indian VPS (Hetzner India, DigitalOcean Bangalore, or AWS Lightsail Mumbai). Add basic auth with streamlit-authenticator. 2. FIX THE COST MODEL — IT'S WRONG BY 40%

The blueprint says ₹2,100-3,350/month. Reality: Supabase Pro ($25) + VPS ($10-20) + Sarvam Vision (₹2,999 in Phase 1) = ₹5,000-7,000/month. Still very cheap! But don't build a business on wrong numbers. Correct the cost table before showing to anyone. Add a "break-even user count" calculation: how many paying users to cover fixed costs?

3. ADD VERSION CONTROL TO REFERENCE ORDERS

Before a single correction replaces a single original, build the version chain. 2 extra columns in the database: version_number and parent_id. "Revert to original" button. This prevents the single most dangerous data corruption path in the entire system. 30 minutes of work that saves months of pain.
